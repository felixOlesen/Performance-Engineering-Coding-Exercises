class mysem {
  public:
    mysem(uint32_t init_value){ 
      counter.store(init_value, std::memory_order_seq_cst); 
    };
    
    void acquire(){
      int current = counter.load(std::memory_order_seq_cst);
    
      while (current == 0) {
        counter.wait(current); 
        current = counter.load(std::memory_order_seq_cst);
      }

      counter.fetch_sub(1, std::memory_order_seq_cst);
    };
    
    void release(){ 
      counter.fetch_add(1, std::memory_order_seq_cst);
      counter.notify_one();
    };
  
  private:
    std::atomic<uint32_t> counter;
};

void acquire(){
  uint32_t* counter_ptr = reinterpret_cast<uint32_t*>(&counter);

  while (true) {
    for(int i=0; i < 100; i++) {

      uint32_t current = counter.load(std::memory_order_seq_cst);
      
      if(current > 0) {
        counter.fetch_sub(1, std::memory_order_seq_cst);
        return;
      }
    }


    syscall(SYS_futex, counter_ptr, FUTEX_WAIT, 0, nullptr, nullptr, 0);
  }
};

void release(){
  uint32_t* counter_ptr = reinterpret_cast<uint32_t*>(&counter);

  counter.fetch_add(1, std::memory_order_seq_cst);

  // Futex wake up one thread
  syscall(SYS_futex, counter_ptr, FUTEX_WAKE, 1, nullptr, nullptr, 0);

};

Question 2.1

Step 1: Calculate the total average message sizes for uploads and downloads.

Uploads:

GET_request_size = 1 Byte (Identifier) + 512 Bytes (Key) = 513 Bytes
SET_request_size = 1 Byte (Identifier) + 512 Bytes (Key) + 1024 Bytes (Value) = 1537 Bytes

Total_avg_upload_size = GET_request_size * 0.9 + SET_request_size * 0.1
                      = 513 * 0.9 + 1537 * 0.1 
                      = 615.4 Bytes/Operation

Downloads:

GET_response_size = 1024 Bytes (Value)
SET_response_size = 1 Byte (ACK)

Total_avg_upload_size = GET_response_size * 0.9 + SET_response_size * 0.1
                      = 1024 * 0.9 + 1 * 0.1 
                      = 921.7 Bytes/Operation

Step 2: Convert 10Gbps to Bytes and determine if downloads or uploads will be the limiting factor.

10Gbps = 1.25 Gigabytes/second
       = 1.25 * 10^9 Bytes/second

As calculated in step 1, the limiting factor for the fully-duplexed network card will be downloads, since it will saturate the network card upload bandwidth faster than with uploads.

Step 3: Calculate Maximum Total Throughput

Max_Total_Throughput = (1.25 * 10^9 Bytes/second) / Total_avg_upload_size
                     = (1.25 * 10^9) / 921.7
                     = 1,356,189.65 Operations/second

Step 1: Convert all variables into similar units.

total_requests_per_second = 130,000 requests/second

GET_request_latency = 200 * 10^-6 seconds
SET_request_latency = 500 * 10^-6 seconds

Assuming that the ratio for requests is still 90% GETs and 10% SETs, the request distribution will be:

GET_request_rate = total_requests_per_second * 0.9
                  = 130,000 * 0.9
                  = 117,000 requests/second


SET_request_rate = total_requests_per_second * 0.1
                  = 130,000 * 0.1
                  = 13,000 requests/second

Step 2: Calculate the work 1 CPU can do in 1 second for each request type

CPU_GET_work = 1 / GET_request_latency
             = 1 / (200 * 10^-6)
             = 5000 GET requests per second

CPU_SET_work = 1 / SET_request_latency 
             = 1 / (500 * 10^-6)
             = 2000 SET requests per second


Step 3: Calculate total number of CPUs required to serve the incoming requests.

GET_CPU_count = GET_request_rate / CPU_GET_work
              = 117,000 / 5000
              = 23.4 CPUs 

SET_CPU_count = SET_request_rate / CPU_SET_work
              = 13,000 / 2000
              = 6.5 CPUs

Total_CPUs_Required = GET_CPU_count + SET_CPU_count
                    = 23.4 + 6.5
                    = 29.9 CPUs
                    = 30 CPUs // Rounded up as you can't have 0.9 of a CPU

Question 2.3:

Experiment 1: 1000 req/seq, 100% GETs

1000 requests per second would be considered a very light load that could be handled easily by 1 CPU for both the lock-free and lock-containing implementations. The overhead due to any locking from the shared lock would be negligibly small, allowing for nearly identical latencies in comparison to the ideal, lock-free implementation. This shows why the average latencies for the ideal and lock implementations are 200 micro seconds.

Experiment 2: 100000 req/seq, 100% GETs

The lock implementation features a slightly longer latency of 207 micro seconds which demonstrates the overhead due to the shared locking mechanism incurred by acquiring and releasing resources, incrementing and decrementing the internal counter of the lock. The additional latency is more apparent due to the increase in traffic by a factor of 100, requiring multiple CPUs to support and incurring more overhead due to locking, in spite of the shared lock allowing for concurrent reads. The ideal implementation would remain at its latency average of 200 micro seconds under this workload since it is lock-free.


Experiment 3: 100000 req/seq, 50% GETs, 50% SETs

In this case, the lock-free implementation would have significantly lower latency than the lock implementation. This would be caused by the inclusion of the 50,000 SET operations per second, which require exclusive locks. This incurs longer latencies due to keeping exclusive access to a resource, blocking reads and writes from other threads. This would block GET operations and other SET operations from acquiring resources until the current thread is done with it, increasing the number of waiting threads at any given time.
